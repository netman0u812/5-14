Executive Summary

The SNA vision for CVS Health

CVS Health operates one of the most architecturally complex enterprise networks in the United States — the product of 15+ years of acquisitions culminating in the 2018 Aetna merger. Two heritage enterprises, four overlapping RFC1918 address spaces, six routing domains, 11,000 retail locations, and a SNAT-masked SD-WAN fabric create a network environment that standard NDR deployment models cannot accommodate without modification.

This document defines the strategic vision and deployment architecture for Cisco Secure Network Analytics (CSNA) as the enterprise-wide Network Detection and Response platform for CVS Health. The model is grounded in validated analysis of three production datasets comprising 326,809 records. The central architectural conclusion is a five-stack independent Manager deployment, each stack representing a distinct routing domain with its own telemetry mechanism, Flow Collector placement, and behavioral baseline scope. Cross-domain threat correlation is delivered by Cisco XDR and Splunk, which operate above the SNA layer using entity-based correlation rather than IP-based host tracking.

When fully deployed, CSNA will provide pervasive NetFlow visibility across CVS Enterprise, Aetna Enterprise, CVS Retail (at the SD-WAN aggregate level), the Switch COLO fabric, and the cloud infrastructure — covering every major routing domain in the CVS Health environment without requiring a single physical traffic tap.

Section 1

The five-stack architecture —
derived from production data, not assumed

The five-stack model is not an arbitrary organizational choice. It is the minimum architecture that correctly resolves CVS Health’s IP overlap constraints. The decision is data-driven: analysis of the production IPAM (v3.78) identified 45 confirmed IP address collisions across the CVS Enterprise and Aetna Enterprise routing domains, and the retail dataset confirmed that store device IPs are permanently SNAT-masked at the store edge. A single SNA Manager with domain isolation would silently corrupt behavioral baselines for all 45 collision IPs from day one. Five independent Manager + Data Store instances eliminate this by construction.

Why five stacks are mandatory, not optional: In SNA, behavioral analytics build per-host baselines keyed on IP address. When the same IP address represents two different physical hosts in different routing domains, a shared Manager creates a composite baseline for a fictional entity. Alerts generated from this baseline are unreliable in both directions — false positives and false negatives. The 45 confirmed collisions include public address space (157.121.x, 206.213.x), RFC1918 (10.x, 172.x), GCP BU allocations, and even documentation space (203.0.113.0). Without five separate Managers, CVS Health cannot produce trustworthy SNA detections for any subnet in the collision inventory.

Cross-stack correlation architecture: Each SNA Manager is an independent data silo for its routing domain. XDR correlates by entity (user identity from ISE, hostname, certificate fingerprint) rather than by IP address, which sidesteps the IP overlap problem entirely at the correlation layer. Splunk, receiving domain-tagged events from all five stacks, provides the historical cross-domain hunting capability.

Primary detection use cases — CVS Health threat landscape

Healthcare data exfiltration
Healthcare is the #1 ransomware target and the #1 data breach sector. CVS holds member PHI (Aetna Stack 2), PBM prescription records (CVS Stack 1), and claims data. SNA behavioral baselining detects bulk transfer anomalies, unusual destination countries, off-hours patterns, and Talos TI matches against known exfiltration infrastructure.
Ransomware pre-deployment detection
SNA detects the reconnaissance and lateral movement patterns that precede encryption events: SMB/RDP volume spikes, port sweep anomalies, worm propagation signatures (MITRE T1570), and service account connection anomalies. Detecting these in Stacks 1 and 2 is the primary ransomware use case — stopping the kill chain before payload deployment.
Encrypted C2 detection (ETA)
Cisco ETA analyzes TLS handshake metadata, certificate chain properties, and beacon timing patterns without decrypting traffic. Critical in a no-tap environment where L7 payload inspection is unavailable. Supplemented by PAN-OS DNS Security and App-ID at firewall insertion points.
Retail store anomaly detection (Stack 3)
Store WAN IP behavioral baselines detect: unusual egress volume (possible POS compromise), atypical destination (possible C2), off-hours communication bursts, and store-to-enterprise tunneling anomalies. Detection operates at the per-store aggregate level — individual device-level visibility is not available due to SNAT.

Section 4

Identity & context layer

Network flow data alone identifies hosts by IP address. The identity layer binds those IP addresses to users, devices, and roles — transforming raw flow anomalies into actionable security events. In a healthcare environment, knowing that an unusual data transfer originates from a specific nurse’s workstation versus an application server is the difference between a true positive and noise.

Cisco ISE pxGrid

User & device identity binding

ISE pxGrid provides real-time user-to-IP mapping from 802.1X authentication, RADIUS accounting, and DHCP. Each SNA Manager connects to a domain-scoped pxGrid subscription — Stack 1 (CVS) Manager must not receive Aetna user-IP mappings, and vice versa. Incorrect cross-domain pxGrid scope is equivalent to the IP collision problem applied to identity.

  • ✓Real-time user-to-IP binding in SNA flows
  • ✓Device type and posture classification
  • ✓ISE → SNA detection → ISE quarantine enforcement loop
  • ✗Not available for Stack 3 Retail (SNAT masks device identity)

PAN-OS User-ID

Firewall-layer identity supplement

PAN-OS firewalls provide User-ID binding at the firewall insertion point, complementing ISE for non-802.1X-authenticated traffic. When PAN-OS IPFIX with App-ID is exported to SNA Flow Collectors, user context from PAN-OS User-ID is included in the flow record, enriching SNA behavioral baselines at CVS and Aetna perimeters.

  • ✓User-IP binding at PAN-OS egress and inter-zone firewall points
  • ✓App-ID provides L7 application context to SNA flows
  • ✓Supplements ISE for guest, contractor, and IoT segments

Encrypted Traffic Analytics (ETA)

TLS threat detection without decryption

ETA is critical in a no-tap environment where L7 payload inspection is not available. ETA analyzes TLS handshake metadata (cipher suite, certificate chain, SNI) and flow behavioral patterns (burst timing, session length distribution) to classify encrypted sessions as benign or malicious without decryption, preserving PHI and member data privacy.

  • ✓TLS certificate fingerprinting against Talos known-bad infrastructure
  • ✓Beacon timing pattern detection in encrypted sessions
  • ✓DGA (domain generation algorithm) detection via DNS flow patterns
  • ✓No PHI exposure — no decryption required
Section 5

Integration architecture —
SNA as the network telemetry hub

SNA does not operate in isolation. Each of the five stacks feeds into a common correlation and response layer. SNA is the network telemetry source; Cisco XDR is the cross-domain correlation engine; Splunk is the long-term retention and SOC hunt platform; ISE is the enforcement layer.

Integration architecture — data flows
Five SNA Managers
→
Each Manager forwards enriched flow records (with host groups, user identity, threat scores, behavioral flags) via syslog/API. Domain tag is preserved on every forwarded event.
Cisco XDR
→
SNA detections from all five stacks feed XDR. XDR correlates by entity — user identity, hostname, certificate fingerprint — not by IP address. A network-based lateral movement indicator is correlated with endpoint (Secure Endpoint) and email signals to form an incident. Cross-stack threat correlation (CVS ↔ Aetna) is viable in XDR because entity-based correlation sidesteps the IP overlap problem.
Splunk SIEM
→
All five stacks forward domain-tagged events to Splunk. Stack context (Stack 1–5) is a queryable field on every event. Provides long-term retention for forensic investigation, cross-domain hunt workflows, and SOC 2 evidence collection. SNA replaces raw NetFlow ingestion in Splunk with structured, enriched, annotated flow data.
Cisco ISE enforcement
→
When SNA detects a compromised host in Stack 1 or Stack 2, ISE pxGrid enables automated quarantine — changing the host’s network access policy in real time. The network becomes self-defending: detection in SNA triggers containment in ISE without manual firewall rule changes. Runbook required: SNA detection severity threshold → automated ISE policy change → SOC notification → analyst review within defined SLA.
Section 6

CCD Platform as the governance foundation —
SNA configuration is derived, never manual

The most common cause of SNA deployment failure in complex environments is host group misconfiguration: subnets assigned to the wrong stack, overlapping CIDRs in host group definitions, or inferred subnet locations that place a CVS subnet in the Aetna Manager. CVS Health’s CCD Platform eliminates this risk by providing three authoritative datasets from which all SNA host group configuration is derived.

all_IP_networks v3.78

Authoritative enterprise subnet registry

20,942 subnets. Every subnet has a verified CANONICAL_LOCATION, ROUTING_DOMAIN, and HERITAGE field. The ROUTING_DOMAIN field is the direct input for stack assignment. The DATA_QUALITY field (Authoritative vs Inferred) drives host group naming — authoritative and inferred subnets go to separate host group tiers, enabling SOC analysts to weight alert context appropriately.

Non-negotiable rule: No subnet enters an SNA host group without a verified CANONICAL_LOCATION in the location_master. Update order: LM → IPAM → SNA configuration. The Observed→Inferred DATA_QUALITY patch (open since v3.71) must be resolved before Stack 1 production baselining begins.

location_master v5.9g

Authoritative site + FC anchor registry

11,526 sites with verified SITE_CODEs, NET_TYPE, HERITAGE, and LOCATION_TYPE. All proposed Flow Collector deployment sites are confirmed present in the LM with correct attributes. The LM is the input for FC host group naming, SITE_CODE-based host group construction, and the retail Store WAN IP → SITE_CODE mapping for Stack 3. One unplanned CVS DC (Wood Dale IL, US_WOI_IL_DC) requires Stack 1 architecture assessment.

retail_networks v2.8

Stack 3 store WAN IP reference

294,341 device records across 9,144 stores. For SNA purposes: the device-level /32 IPs are store-internal and irrelevant (SNAT-masked). The dataset provides the canonical SITE_CODE, physical address, and store metadata required to name and document Stack 3 host groups. The store WAN/tunnel IPs (SD-WAN WAN-side interface addresses) are the actual host identifiers in Stack 3 — these must be sourced from the 128T SD-WAN controller inventory, not retail_networks.

CCD Platform governance rule for SNA configuration updates: When any CCD Platform dataset is updated (IPAM version increment, new LM entries, retail store additions), the SNA host group configuration for affected stacks must be reviewed and updated before the next IPAM cross-check cycle. The stack assignment for any subnet is governed by its ROUTING_DOMAIN value in all_IP_networks — this is not a field that can be overridden by server owner, application team, or manual estimate. Any proposed IPAM update that changes a subnet’s ROUTING_DOMAIN must be flagged for SNA host group review as part of the change governance process.
